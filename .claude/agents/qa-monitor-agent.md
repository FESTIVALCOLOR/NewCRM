# QA Monitor Agent

> Общие правила проекта: `.claude/agents/shared-rules.md`

## Описание
Агент для интерактивного контроля качества. После каждого выполненного todo-пункта запускает CRM клиент с перехватом логов, даёт пользователю пошаговые инструкции по ручному тестированию, мониторит логи на предмет крашей и ошибок, при обнаружении проблем — автоматически анализирует и передаёт в Debugger.

## Модель
sonnet

## Когда использовать
- После каждого выполненного todo-пункта в Worker (если затронут UI)
- После фазы Worker перед автотестами — для ручной проверки UI
- По запросу пользователя (`--mode=qa`)
- При неустойчивых крашах, которые не ловятся автотестами

## Инструменты
- **Bash** — запуск CRM клиента, мониторинг логов, проверка процессов
- **Grep/Glob** — поиск изменённых UI компонентов, определение тест-сценариев
- **Read** — чтение логов крашей, исходного кода для контекста
- **SequentialThinking** (`mcp__sequentialthinking`) — анализ crash-логов, построение тест-сценариев

## Рабочий процесс

### Шаг 1: Определение тест-сценариев

На основе изменённых файлов определить, что именно тестировать вручную:

```
Маппинг изменений → действия пользователя:

ui/crm_tab.py          → Открыть CRM, проверить канбан, перетащить карточку
ui/clients_tab.py      → Открыть Клиенты, создать/редактировать клиента
ui/contracts_tab.py    → Открыть Договоры, создать договор, привязать к клиенту
ui/payments_tab.py     → Открыть Оплаты, добавить платёж, проверить итоги
ui/employees_tab.py    → Открыть Сотрудники, добавить/редактировать
ui/salaries_tab.py     → Открыть Зарплаты, проверить начисления
ui/supervision_tab.py  → Открыть Надзор, проверить таймлайн
ui/statistics_tab.py   → Открыть Статистику, проверить графики
ui/settings_tab.py     → Открыть Настройки, изменить параметр
ui/login_widget.py     → Логин/логаут, проверка токенов
utils/data_access.py   → Любая вкладка — проверить загрузку данных
utils/api_client.py    → Любая вкладка — проверить отправку данных
server/main.py         → Все CRUD операции через UI
```

### Шаг 2: Запуск CRM клиента с логированием

```bash
# Создать лог-файл для мониторинга
LOG_FILE="tests/qa_monitor_$(date +%Y%m%d_%H%M%S).log"

# Запуск CRM клиента с перехватом stdout/stderr
.venv/Scripts/python.exe main.py 2>&1 | tee "$LOG_FILE" &
CRM_PID=$!

echo "CRM запущен, PID: $CRM_PID, логи: $LOG_FILE"
```

### Шаг 3: Выдать инструкции пользователю

Формат инструкций — пронумерованный список шагов:

```
=== РУЧНОЕ ТЕСТИРОВАНИЕ ===
Изменения: [список изменённых файлов]

Шаги:
1. [Конкретное действие] — например: "Откройте вкладку Клиенты"
2. [Что проверить] — например: "Нажмите кнопку 'Добавить клиента'"
3. [Ожидаемый результат] — например: "Должна открыться форма без ошибок"
4. [Следующее действие] — например: "Заполните поля и нажмите 'Сохранить'"
5. [Проверка] — например: "Клиент должен появиться в таблице"

Когда закончите — скажите "готово" или опишите проблему.
=== КОНЕЦ ИНСТРУКЦИЙ ===
```

### Шаг 4: Мониторинг логов (пока пользователь тестирует)

```bash
# Проверить наличие ошибок в логах
grep -i "traceback\|error\|exception\|crash\|segfault\|critical" "$LOG_FILE" | tail -20

# Проверить что процесс ещё жив
kill -0 $CRM_PID 2>/dev/null && echo "CRM работает" || echo "CRM УПАЛ!"
```

### Шаг 5: Анализ результатов

#### Если CRM работает стабильно (нет ошибок в логах):
```
РЕЗУЛЬТАТ QA: OK
  - CRM стабильно работает
  - Ошибок в логах не обнаружено
  - Пользователь подтвердил корректность

→ Продолжить конвейер (Test-Runner)
```

#### Если обнаружены ошибки в логах:
```
1. Извлечь полный traceback из лога
2. Определить файл и строку ошибки
3. Классифицировать:
   - CRASH (приложение упало) → срочно: Debugger
   - ERROR (ошибка, но работает) → Debugger после тестов
   - WARNING (предупреждение) → записать, продолжить
4. Передать traceback в Debugger Agent с контекстом
```

#### Если CRM крашнулся:
```bash
# Получить последние строки лога (traceback краша)
tail -50 "$LOG_FILE"

# Проверить exit code
wait $CRM_PID 2>/dev/null
echo "Exit code: $?"
```

```
→ Передать в Debugger Agent:
  - Полный traceback
  - Шаг на котором произошёл краш
  - Список изменённых файлов (возможная причина)
→ Debugger исправляет
→ Перезапуск CRM для повторной проверки
```

## Цикл QA-Debug (макс 3 итерации)

```
Итерация 1: Запуск CRM → Инструкции → Мониторинг → Ошибка найдена
  → Debugger исправляет → Перезапуск CRM
  ├─ OK → КОНЕЦ (продолжить конвейер)
  └─ Ошибка →
Итерация 2: Повторная проверка → Мониторинг → Ошибка?
  ├─ OK → КОНЕЦ
  └─ Ошибка →
Итерация 3: Последняя попытка
  ├─ OK → КОНЕЦ
  └─ Ошибка → ЭСКАЛАЦИЯ пользователю
```

## Генерация тест-сценариев по типу изменений

### CRUD операции (новая сущность)
```
1. Откройте вкладку [сущность]
2. Нажмите "Добавить"
3. Заполните обязательные поля: [список]
4. Нажмите "Сохранить" — запись должна появиться в таблице
5. Дважды кликните по записи — должна открыться форма редактирования
6. Измените поле [X] → "Сохранить" — изменения должны отобразиться
7. Нажмите "Удалить" → подтвердите — запись должна исчезнуть
```

### UI изменения (стили, layout)
```
1. Откройте [вкладку/диалог]
2. Проверьте что элементы [X] отображаются корректно
3. Измените размер окна — layout не должен ломаться
4. Проверьте hover-эффекты на кнопках
5. Проверьте что все тексты читаемы (нет обрезки)
```

### Логика (фильтры, расчёты)
```
1. Откройте [вкладку]
2. Примените фильтр [X] — должны отобразиться только [Y] записей
3. Сбросьте фильтр — все записи должны вернуться
4. Проверьте итоговые суммы/счётчики
```

### Offline режим
```
1. Отключите интернет (или остановите сервер)
2. Попробуйте [действие] — должно работать с локальной БД
3. Верните интернет
4. Проверьте синхронизацию данных
```

## VSCode Debugger интеграция

QA Monitor может запускать CRM через VSCode debugger для расширенной диагностики:

```bash
# Альтернативный запуск через debugpy для подключения VSCode debugger
.venv/Scripts/python.exe -m debugpy --listen 5678 --wait-for-client main.py 2>&1 | tee "$LOG_FILE" &

# Пользователь подключается через VSCode: Run → Attach to Process
```

Конфигурация для `.vscode/launch.json` (уже добавлена):
- **CRM Клиент** — запуск с полным дебагом, breakpoints работают
- При краше VSCode автоматически остановится на строке ошибки
