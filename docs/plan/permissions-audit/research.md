## 1. АРХИТЕКТУРА СИСТЕМЫ ПРАВ — ТЕКУЩЕЕ СОСТОЯНИЕ

### 1.1 Три уровня проверки прав

Система прав в проекте реализована на **трёх уровнях**, которые работают **несогласованно**:

| Уровень | Где | Как работает | Проблема |
|---------|-----|-------------|----------|
| **1. config.py ROLES** | `main_window.py` | Жёсткий словарь `ROLES[position]['tabs']` определяет видимость вкладок | Не связан с permission system |
| **2. `_emp_has_pos()`** | `crm_tab.py`, `crm_card_edit_dialog.py` и др. | Проверка должности сотрудника, hardcoded списки | Не связан с permission system |
| **3. `_has_perm()` + `require_permission()`** | API endpoints + частично UI | Настраиваемые права из БД/матрицы | **Единственный правильный уровень**, но используется не везде |

### 1.2 Два контура защиты

```
UI (клиент)                              API (сервер)
┌──────────────────────┐                ┌──────────────────────┐
│ config.py ROLES      │  ← ЖЁСТКИЙ    │                      │
│ └─ видимость вкладок │                │                      │
│                      │                │                      │
│ _emp_has_pos()       │  ← ЖЁСТКИЙ    │                      │
│ └─ видимость кнопок  │                │                      │
│                      │                │                      │
│ _has_perm()          │  ← ГИБКИЙ     │ require_permission() │
│ └─ только 5 прав!    │                │ └─ все 38 прав       │
└──────────────────────┘                └──────────────────────┘
```

**Критический факт:** В UI через `_has_perm()` проверяются только ~5 прав:
- `crm_cards.update`
- `contracts.update`
- `messenger.create_chat`
- `messenger.view_chat`
- `messenger.delete_chat`

Все остальные 33 права проверяются ТОЛЬКО на сервере, а UI скрывает/показывает элементы по ДОЛЖНОСТИ.

## 3. ОТВЕТЫ НА ВСЕ ВОПРОСЫ ПОЛЬЗОВАТЕЛЯ

### 3.1 Отсутствующие роли в таблице

**Проблема подтверждена.** В таблице прав отсутствуют:
- **Дизайнер** — есть в `config.py POSITIONS`, есть в `ROLES`, но **нет в таблице прав**
- **Чертёжник** — аналогично
- **Замерщик** — аналогично

Это означает, что для этих трёх ролей **нельзя настроить права** через UI. Они получают пустой набор прав (по умолчанию — ничего), но при этом видят вкладку СРМ по config.py ROLES.

### 3.2 Отсутствие прав доступа к страницам

**Проблема подтверждена.** Доступ к страницам (вкладкам) управляется через `config.py ROLES`, а **НЕ через систему прав**.

| Страница | Как определяется доступ | Через permissions? |
|----------|----------------------|-------------------|
| Клиенты | `ROLES[position]['tabs']` | **НЕТ** |
| Договора | `ROLES[position]['tabs']` | **НЕТ** |
| СРМ | `ROLES[position]['tabs']` | **НЕТ** |
| СРМ надзора | `ROLES[position]['tabs']` | **НЕТ** |
| Отчёты и Статистика | `ROLES[position]['tabs']` | **НЕТ** |
| Сотрудники | `ROLES[position]['tabs']` | **НЕТ** |
| Зарплаты | `ROLES[position]['tabs']` | **НЕТ** |
| Отчёты по сотрудникам | `ROLES[position]['tabs']` | **НЕТ** |

**Вывод:** Нет возможности через администрирование настроить, какие страницы видит сотрудник. Это захардкожено.

### 3.3 Отсутствие системного разделения CRUD для каждой страницы

**Проблема подтверждена.** Текущие права покрывают CRUD крайне неравномерно:

| Страница | Создание | Просмотр | Редактирование | Удаление |
|----------|---------|----------|---------------|---------|
| **Клиенты** | НЕТ права | НЕТ права | НЕТ права | `clients.delete` |
| **Договора** | НЕТ права | НЕТ права | `contracts.update` | `contracts.delete` |
| **CRM** | НЕТ права | НЕТ права | `crm_cards.update` | `crm_cards.delete` |
| **Надзор** | НЕТ права | НЕТ права | `supervision.update` | `supervision.delete_order` |
| **Сотрудники** | `employees.create` | НЕТ права | `employees.update` | `employees.delete` |
| **Платежи** | `payments.create` | НЕТ права | `payments.update` | `payments.delete` |
| **Зарплаты** | `salaries.create` | НЕТ права | `salaries.update` | `salaries.delete` |

**Нет прав на:**
- **Создание** клиентов, договоров, CRM карточек, карточек надзора
- **Просмотр** чего-либо — все данные видны всем, у кого есть доступ к вкладке

### 3.4 Управление внутри карточки (файлы, оплаты)

**Проблема подтверждена.** Нет прав на:
- Загрузка файлов — контроль только по должности (`is_only_designer`)
- Удаление файлов — контроль только по должности (`can_delete`)
- Редактирование оплат — вкладка скрыта по должности, нет проверки `_has_perm()`
- Удаление оплат — аналогично

## 4. ОТВЕТЫ НА ВОПРОСЫ ПО CRM

### 4.1 Удаление исполнителей

**Где управляется:** На сервере есть endpoint `DELETE /api/crm/stage-executors/{executor_id}` с правом `crm_cards.delete_executor`.
**Кнопка в UI:** Отдельной кнопки "Удалить исполнителя" **НЕТ**. Есть только кнопка "Переназначить" (иконка refresh) в `crm_card_edit_dialog.py:693, 816`. При переназначении старый исполнитель заменяется новым, удаление как таковое не происходит через UI.

### 4.2 Сброс стадий CRM, сброс согласования, завершение согласования, сброс отметки дизайнера/чертежника

| Действие | Где в коде | Кнопка | Как срабатывает |
|----------|-----------|--------|----------------|
| **Сброс стадий** | `crm_tab.py:767-785` | **Нет кнопки** — автоматически при перемещении карточки между колонками (drag-drop) | Вызывается `data.reset_stage_completion(card_id)` |
| **Сброс согласования** | `crm_tab.py:770` | **Нет кнопки** — автоматически при возврате карточки из архива | Вызывается `data.reset_approval_stages(card_id)` |
| **Завершение согласования** | Серверный endpoint `/api/crm/cards/{id}/complete-approval` | В UI вызывается при финальном согласовании проекта | Право `crm_cards.complete_approval` |
| **Сброс отметки дизайнера** | Серверный endpoint `/api/crm/cards/{id}/reset-designer` | В UI привязан к определённым действиям менеджера | Право `crm_cards.reset_designer` |
| **Сброс отметки чертежника** | Аналогично | Аналогично | Право `crm_cards.reset_draftsman` |

**Вывод:** Эти действия происходят **неявно** — при перемещении карточек. Отдельных кнопок нет. Права на сервере проверяются, но в UI отдельного контроля нет.

### 4.3 Доступ ко вкладкам карточки CRM

| Вкладка | Кто видит | Проверка |
|---------|----------|----------|
| Исполнители и дедлайн | Все | Без ограничений |
| Таблица сроков | Все | Без ограничений |
| Данные по проекту | Все | Без ограничений |
| **История по проекту** | **Не исполнители** (скрыта для дизайнеров/чертёжников) | `_emp_has_pos()` |
| **Оплаты** | **Только менеджеры+** (РС, СМП, СДП, ГАП, Менеджер) | `_emp_has_pos()` |

**Права через permission system НЕ используются** для вкладок.

### 4.4 Файлы, оплаты, переназначение, дедлайны в карточке CRM

| Функция | Проверка в UI | Проверка на сервере | Через permissions? |
|---------|-------------|--------------------|--------------------|
| Загрузка файлов | По должности | Только `get_current_user` | **НЕТ** |
| Удаление файлов | По должности | Только `get_current_user` | **НЕТ** |
| Редактирование оплат | Видимость вкладки по должности | `payments.update` | Частично (сервер) |
| Удаление оплат | Видимость вкладки по должности | `payments.delete` | Частично (сервер) |
| Переназначение исполнителей | `has_full_access` (должность) | `crm_cards.assign_executor` | Частично (сервер) |
| Установка дедлайнов | `has_full_access` (должность) | `supervision.update` | Частично (сервер) |

**Права `payments.*` сейчас глобальные** — нет разделения на "оплаты в CRM" и "оплаты в зарплатах".

## 7. ОТВЕТЫ НА ОСТАЛЬНЫЕ ВОПРОСЫ

### 7.1 Ставки — что это?

**Ставки = Тарифы.** Это тарифные ставки за различные виды работ:
- Индивидуальные проекты (цена за м²)
- Шаблонные проекты (диапазоны площадей)
- Авторский надзор (цена за м²)
- Замерщик (по городам: СПБ, МСК, ВН)

В UI это `RatesDialog` в AdminDialog, вкладка "Тарифы". Название "Ставки" в permissions — это те же самые тарифы.

### 7.2 Агенты и города — зачем в правах?

**Проблема подтверждена.** Агенты и города управляются ТОЛЬКО из AdminDialog (вкладка "Агенты и города"), доступ к которому имеет только Руководитель/admin/director. Наличие прав `agents.*` и `cities.*` в матрице — **избыточно**, потому что:
1. Кнопка "Администрирование" видна только суперпользователям
2. Суперпользователи и так имеют все права

**Единственный смысл** — если в будущем доступ к AdminDialog будет расширен на другие роли.

### 7.3 Мессенджер — отправка скриптов

**Скрипты** в мессенджере — это **шаблоны автоматических сообщений** (Telegram, Email) с плейсхолдерами (`{{client_name}}`, `{{project_address}}` и т.д.). Управление скриптами находится в `MessengerAdminDialog`, вкладка "Скрипты".

**Прав на управление скриптами НЕТ** — доступ только через AdminDialog (суперпользователи). Текущие права мессенджера:
- `messenger.create_chat` — создание чатов
- `messenger.delete_chat` — удаление чатов
- `messenger.view_chat` — просмотр чатов

Нет `messenger.manage_scripts`.

### 7.4 Администрирование — доступ

**Проблема подтверждена.** Доступ к диалогу администрирования проверяется по ДОЛЖНОСТИ в `employees_tab.py:801`:
```python
if current_role in ('admin', 'director') or current_pos == 'Руководитель студии':
```
Нет права `admin.access` в системе permissions.

### 7.5 Глобальный поиск и кнопки

**Глобальный поиск:** Сервер (`main.py:253`) возвращает ВСЕ результаты без фильтрации по правам. Если у роли нет доступа к клиентам — поиск всё равно покажет клиентов.

**Кнопки на страницах (обновить БД, статистика и т.д.):** Нет проверки прав на показ этих кнопок. Видимость определяется наличием вкладки (config.py ROLES).
