## 1. Общее описание

### Что это?

Система аналитики по сотрудникам — это расширение страницы «Отчёты по сотрудникам» в CRM Interior Studio. Вместо текущих двух простых таблиц (выполненные заказы + зарплаты) мы создаём полноценную аналитическую платформу, которая позволяет:

- **Руководству** — видеть общую картину эффективности команды на одном экране
- **Старшим менеджерам** — сравнивать сотрудников своих команд между собой
- **Самим сотрудникам** — понимать свои сильные и слабые стороны

### Какую проблему решаем?

Сейчас руководитель студии не имеет единого места, где можно увидеть:
- Кто из сотрудников работает хорошо, а кто — нет
- Кто систематически срывает сроки
- Сколько проектов сотрудник может вести параллельно без потери качества
- Как клиенты оценивают работу конкретного сотрудника
- Какие тенденции в работе сотрудников (улучшается / ухудшается)

### Три типа проектов — три отдельных потока аналитики

В Interior Studio есть три принципиально разных типа проектов. У каждого своя структура, свои стадии, свои участники и свои метрики. **Смешивать их в одном графике нельзя** — это как сравнивать метры с килограммами.

| Тип проекта | Стадии | Участники | Что измеряем |
|-------------|--------|-----------|-------------|
| **Индивидуальный** | Смета → Замер → Планировка → Концепция → Рабочий проект | СМ, СДП, ГАП, Чертёжники, Дизайнеры, Замерщики | Скорость стадий, качество, правки |
| **Шаблонный** | Планировка → Рабочие чертежи → 3Д визуализация | СМ, Менеджеры, ГАП, Чертёжники, Дизайнеры | Скорость стадий, качество, правки |
| **Авторский надзор** | 12 стадий закупок | СМ, ДАН | Дефекты, визиты, бюджет, сроки |

## 2. Текущее состояние

### Что уже есть

Страница **«Отчёты по сотрудникам»** (`ui/employee_reports_tab.py`, 474 строки) содержит:

**3 вкладки:** Индивидуальные проекты | Шаблонные проекты | Авторский надзор

В каждой вкладке:
- Фильтры по периоду (год, квартал, месяц)
- Таблица «Выполненные заказы» (ФИО, должность, количество)
- Таблица «Сумма зарплаты» (ФИО, должность, сумма)
- Кнопки экспорта в PDF

**Серверный API** (`server/routers/reports_router.py`, 329 строк):
- `GET /api/reports/employee` — данные по одному сотруднику
- `GET /api/reports/employee-report` — данные по всем сотрудникам (completed, area, deadlines, salaries)

### Что плохо

1. **Нет KPI** — только голые цифры без интерпретации (хорошо это или плохо?)
2. **Нет сравнения** — нельзя сравнить сотрудников одной роли
3. **Нет трендов** — нельзя увидеть динамику по месяцам
4. **Нет детализации** — нельзя провалиться в конкретного сотрудника
5. **Нет качества** — только количество, без оценки качества работы
6. **Нет обратной связи от клиентов** — субъективная оценка вообще не собирается
7. **Нет нагрузки** — непонятно, сколько проектов сотрудник ведёт одновременно
8. **Нет разграничения доступа** — все видят одно и то же

## 4. Разделение по типам проектов

### Почему нельзя смешивать?

Пример из жизни: Чертёжник Иванов делает 5 стадий по индивидуальным проектам (сложные, большая площадь, долгие) и 12 стадий по шаблонным (простые, маленькая площадь, быстрые). Если посчитать общий KPI по всем — получится «средняя температура по больнице», которая ничего не скажет ни о его работе на индивидуальных, ни на шаблонных.

### Как работает разделение

```
Пользователь выбирает вкладку "Индивидуальные проекты"
    ↓
Сервер фильтрует данные:
    - CRM-карточки: JOIN contracts WHERE project_type = 'Индивидуальный'
    - StageExecutor: только по отфильтрованным CRM-карточкам
    - Payments: только по отфильтрованным контрактам
    - Опросы: только по отфильтрованным контрактам
    ↓
KPI считается ТОЛЬКО по данным индивидуальных проектов
    ↓
Графики и таблицы показывают ТОЛЬКО индивидуальные проекты
```

### Какие роли участвуют в каждом типе

Это критически важно: **вкладки ролей меняются** в зависимости от выбранного типа проекта, потому что в разных типах проектов участвуют разные роли.

#### Индивидуальные проекты

```
Стадии workflow:
  Смета → Замер → Планировка → Концепция → Согласование → Рабочий проект

Участники и их роли:
┌─────────────────────────────┬──────────────────────────────────────────────┐
│ Роль                        │ Что делает                                   │
├─────────────────────────────┼──────────────────────────────────────────────┤
│ Старший менеджер            │ Куратор проекта, контролирует сроки          │
│ СДП (Ст.дизайнер-проект.)  │ Проверяет стадии 1-2 (Планировка, Концепция)│
│ ГАП (Гл.архитектор-проект.)│ Проверяет стадию 3 (Рабочий проект)         │
│ Чертёжник                   │ Исполнитель: чертежи планировки, рабочие    │
│ Дизайнер                    │ Исполнитель: концепция дизайна               │
│ Замерщик                    │ Выезд на объект, обмеры                      │
└─────────────────────────────┴──────────────────────────────────────────────┘

Вкладки ролей: [Ст.менеджеры] [СДП] [ГАП] [Чертёжники] [Дизайнеры] [Замерщики]
```

#### Шаблонные проекты

### KPI считается отдельно для каждого типа

Если сотрудник работает и на индивидуальных, и на шаблонных проектах, у него будет:
- `KPI_индивидуальные = 85%` (считается по стадиям индивидуальных проектов)
- `KPI_шаблонные = 72%` (считается по стадиям шаблонных проектов)

Эти два KPI **нигде не смешиваются** — они показываются на разных вкладках.

### Специфичные метрики по типам проектов

#### Метрики для индивидуальных проектов

| Метрика | Описание | Формула |
|---------|----------|---------|
| Выполнение в срок | % стадий завершённых до дедлайна | `on_time / total * 100` |
| Качество (без правок) | % стадий принятых с первого раза | `no_revision / total * 100` |
| Средняя просрочка | Среднее отклонение от дедлайна (дни) | `AVG(completed_date - deadline)` |
| Площадь обработана | Суммарная площадь (м²) | `SUM(contracts.area)` |
| Правки (revision_count) | Среднее кол-во правок на стадию | `AVG(revision_count)` |
| Нагрузка | Одновременных проектов | Подсчёт активных на дату |

#### Метрики для шаблонных проектов

| Метрика | Описание | Формула |
|---------|----------|---------|
| Выполнение в срок | Аналогично | Аналогично |
| Качество (без правок) | Аналогично | Аналогично |
| Средняя просрочка | Аналогично | Аналогично |
| Кол-во этажей | Суммарно (для шаблонных важно) | `SUM(contracts.floors)` |
| Скорость (дни на стадию) | Среднее время выполнения стадии | `AVG(completed_date - assigned_date)` |
| Нагрузка | Одновременных проектов | Подсчёт активных на дату |

#### Метрики для авторского надзора

Авторский надзор имеет **совершенно другую структуру** — здесь нет StageExecutor'ов с дедлайнами. Вместо этого:

| Метрика | Описание | Источник |
|---------|----------|----------|
| Надзоров ведёт | Кол-во активных карточек надзора | `supervision_cards` |
| Сданных надзоров | Кол-во завершённых за период | `supervision_cards.column_name = 'СДАН'` |
| Визитов на объекты | Кол-во выездов за период | `supervision_visits` |
| Дефектов найдено | Суммарное кол-во | `supervision_timeline_entries.defects_found` |
| Дефектов устранено | Суммарное кол-во | `supervision_timeline_entries.defects_resolved` |
| % устранения дефектов | Качество | `resolved / found * 100` |
| Выполнение по plan_date | Стадии закупок в срок | `actual_date <= plan_date` |
| Экономия бюджета | Сумма | `SUM(budget_savings)` |
| Просрочки закупок | Кол-во стадий с `actual_date > plan_date` | `supervision_timeline_entries` |

## 5. Права доступа

### Принцип: каждый видит только то, что ему нужно для работы

Полный доступ ко всей аналитике имеет **только Руководитель студии**. Все остальные видят урезанную версию, ограниченную их зоной ответственности.

### Что значит «свои команды» для каждой роли

#### Руководитель студии (`Руководитель студии`)
- Видит **всех** сотрудников, все проекты, все метрики
- Полный доступ к зарплатам, NPS, сравнительным таблицам
- Единственный, кто видит Топ-5 лучших/проблемных по всей компании

#### Старший менеджер (`Старший менеджер проектов`)
- Видит сотрудников **только тех проектов, где он назначен куратором**
- Фильтр: `crm_cards.senior_manager_id = current_user.id`
- Из этих проектов извлекаются все исполнители (СДП, ГАП, Менеджеры, Чертёжники, Дизайнеры, Замерщики, ДАН)
- Видит зарплаты **своей команды**
- Видит NPS **своих проектов**
- **НЕ видит** сотрудников других старших менеджеров

```python
# Пример: какие сотрудники видны старшему менеджеру
my_cards = db.query(CRMCard).filter(
    CRMCard.senior_manager_id == current_user.id
).all()

visible_employees = set()
for card in my_cards:
    if card.sdp_id: visible_employees.add(card.sdp_id)
    if card.gap_id: visible_employees.add(card.gap_id)
    if card.manager_id: visible_employees.add(card.manager_id)
    if card.surveyor_id: visible_employees.add(card.surveyor_id)
    for executor in card.stage_executors:
        if executor.executor_id:
            visible_employees.add(executor.executor_id)

# Для надзора аналогично
my_supervision = db.query(SupervisionCard).filter(
    SupervisionCard.senior_manager_id == current_user.id
).all()
for sv in my_supervision:
    if sv.dan_id: visible_employees.add(sv.dan_id)
```

#### СДП (`СДП`)
- Видит **только исполнителей** (чертёжников и дизайнеров) на проектах, где он назначен СДП
- Фильтр: `crm_cards.sdp_id = current_user.id` → `stage_executors.executor_id`
- Видит их KPI, качество, сроки
- **НЕ видит** зарплаты других
- Видит **свою собственную** карточку с полной статистикой

```python
# Какие исполнители видны СДП
my_cards = db.query(CRMCard).filter(CRMCard.sdp_id == current_user.id).all()
visible_executors = set()
for card in my_cards:
    for executor in card.stage_executors:
        if executor.executor_id:
            visible_executors.add(executor.executor_id)
# + свой собственный профиль
visible_executors.add(current_user.id)
```

#### ГАП (`ГАП`)
- Аналогично СДП, но по `crm_cards.gap_id`
- Видит исполнителей на стадиях, которые он проверяет
- Видит свою собственную карточку

#### Менеджер (`Менеджер`)
- Видит исполнителей на проектах, где он назначен менеджером
- Фильтр: `crm_cards.manager_id = current_user.id`
- Видит свою собственную карточку

#### Исполнители (Чертёжник, Дизайнер) и Замерщик
- Видят **только свою собственную** карточку с личной статистикой
- **НЕ видят** сводный дашборд
- **НЕ видят** сравнительные таблицы
- **НЕ видят** других сотрудников
- Видят свои проекты, свои метрики, свои отзывы клиентов, свою зарплату

#### ДАН (`ДАН` / `Дизайнер авторского надзора`)
- Видит **только свою** карточку на вкладке «Авторский надзор»
- Видит свои надзоры, визиты, дефекты

### Как это реализуется на сервере

API endpoint принимает `current_user` из JWT токена и фильтрует данные:

```python
@router.get("/dashboard")
async def get_analytics_dashboard(
    project_type: str,  # 'individual' / 'template' / 'supervision'
    year: int = None,
    current_user: Employee = Depends(get_current_user),
    db: Session = Depends(get_db)
):
    position = current_user.position

    if position == POSITION_STUDIO_DIRECTOR:
        # Полный доступ — все сотрудники
        visible_employee_ids = None  # None = без фильтра
    elif position == POSITION_SENIOR_MANAGER:
        # Только сотрудники своих проектов
        visible_employee_ids = get_team_employee_ids(
            db, current_user.id, project_type, role='senior_manager'
        )
    elif position in (POSITION_SDP, POSITION_GAP, POSITION_MANAGER):
        # Только свои исполнители + свой профиль
        visible_employee_ids = get_team_employee_ids(
            db, current_user.id, project_type, role=position
        )
        visible_employee_ids.add(current_user.id)
    else:
        # Исполнители, замерщики, ДАН — только свой профиль
        visible_employee_ids = {current_user.id}

    return build_dashboard(db, project_type, year, visible_employee_ids)
```

```

2. Выбирает тип проекта: Индивидуальные / Шаблонные / Авторский надзор

4. Сервер:
   a. Определяет роль текущего пользователя (из JWT)
   b. Вычисляет список видимых сотрудников (access_filter.py)
   c. Фильтрует данные по типу проекта

6. При выборе конкретного сотрудника:
   a. GET /api/employee-analytics/{employee_id}/detail?project_type=individual
   b. Сервер проверяет: имеет ли текущий пользователь доступ к этому сотруднику
   c. Если да — возвращает детальные метрики
   d. Если нет — 403 Forbidden
```

## 7. Модуль 1: Сводный дашборд KPI

### Кто видит дашборд

| Роль | Что видит в дашборде |
|------|---------------------|
| Руководитель студии | KPI всех сотрудников, работающих на выбранном типе проектов |
| Старший менеджер | KPI только своей команды на выбранном типе проектов |
| СДП / ГАП / Менеджер | Мини-дашборд: только краткая сводка по своим исполнителям |
| Исполнители / Замерщики / ДАН | Дашборд не показывается — сразу карточка «Моя статистика» |

### KPI-карточки (верхний ряд)

| # | Карточка | Значение | Откуда берём | Формула |
|---|----------|----------|--------------|---------|
| 1 | **Сотрудников** | Число | `employees` | Кол-во сотрудников, работающих на данном типе проектов (с учётом прав доступа) |
| 2 | **Средний KPI** | Процент | Расчёт | Среднее KPI всех видимых сотрудников **по данному типу проекта** |
| 3 | **Выполнение в срок** | Процент | `stage_executors` / `supervision_timeline_entries` | Зависит от типа проекта (см. раздел 4) |
| 4 | **Средняя нагрузка** | Число | `stage_executors` / `supervision_cards` | Среднее кол-во активных проектов/стадий данного типа |
| 5 | **NPS клиентов** | Оценка | `client_surveys` | Средний NPS по опросам **данного типа проектов** |
| 6 | **Активных проектов** | Число | `crm_cards` / `supervision_cards` | Кол-во активных проектов данного типа |

### Топ-5 лучших / Топ-5 проблемных

**Видны только:** Руководителю студии (все) и Старшему менеджеру (только свои).

**Топ-5 лучших** (зелёный акцент):
- Сортировка: по убыванию KPI **данного типа проекта**
- Показываем: ФИО, должность, KPI%, стрелка тренда (вверх/вниз/без изменений)

**Топ-5 проблемных** (красный акцент):
- Сортировка: по возрастанию KPI
- Фильтр: только KPI < 60% ИЛИ просрочки > 3
- Показываем: ФИО, должность, KPI%, кол-во просрочек

### График тренда KPI

Линейный график за последние 6 месяцев:
- Ось X: месяцы
- Ось Y: средний KPI (%)
- Линия: средний KPI по видимым сотрудникам **данного типа проекта**
- Зона: от минимального до максимального KPI (заливка с прозрачностью)

## 8. Модуль 2: Вкладки по ролям

### Какие вкладки показываются

Вкладки **динамические** — они зависят от выбранного типа проекта:

| Тип проекта | Вкладки |
|-------------|---------|
| Индивидуальные | Ст.менеджеры, СДП, ГАП, Чертёжники, Дизайнеры, Замерщики |
| Шаблонные | Ст.менеджеры, Менеджеры, ГАП, Чертёжники, Дизайнеры |
| Авторский надзор | Ст.менеджеры, ДАН |

Плюс, вкладки фильтруются по правам доступа (раздел 5).

### Сравнительная таблица роли

Для каждой роли — таблица со всеми **видимыми** сотрудниками этой роли.

#### Универсальные столбцы (для CRM-проектов: индивидуальных и шаблонных)

| Столбец | Описание | Как считается |
|---------|----------|--------------|
| **ФИО** | Имя сотрудника | `employees.full_name` |
| **KPI** | Интегральная оценка 0-100% | По формуле из раздела 11 |
| **Выполнение в срок** | % стадий, завершённых до дедлайна | `completed_on_time / total_completed * 100` |
| **Просрочки** | Кол-во просроченных стадий за период | `COUNT(completed_date > deadline)` |
| **Ср. просрочка (дн.)** | Среднее кол-во дней просрочки | `AVG(completed_date - deadline)` |
| **Зарплата** | Сумма за период (только для Руководителя и СМ) | `SUM(payments.final_amount)` |
| **Тренд** | Стрелка вверх/вниз/без изменений | Сравнение KPI с предыдущим периодом |

#### Специфичные столбцы по ролям

**Старшие менеджеры (индивидуальные + шаблонные):**

| Столбец | Описание |
|---------|----------|
| Проектов в работе | CRM-карточки с `senior_manager_id` (активные, данного типа) |
| Сданных проектов | Карточки «СДАН» за период |
| Расторгнутых | Карточки «РАСТОРГНУТ» за период |
| Общая стоимость | `SUM(contracts.total_amount)` |
| NPS клиентов | Средний NPS по проектам |

**Старшие менеджеры (авторский надзор):**

| Столбец | Описание |
|---------|----------|
| Надзоров ведёт | `supervision_cards.senior_manager_id` (активные) |
| Сданных надзоров | Завершённые за период |
| Общая экономия | `SUM(budget_savings)` |
| Визитов на объекты | Суммарно по его надзорам |

**СДП (только индивидуальные):**

| Столбец | Описание |
|---------|----------|
| Проверок проведено | Стадии, где СДП выступал reviewer'ом |
| Ср. время ревью (дн.) | От `submitted_date` до действия СДП |
| Возвратов на доработку | Кол-во `revision_count > 0` |
| Проектов | CRM-карточки с `sdp_id` |

**Менеджеры (только шаблонные):**

| Столбец | Описание |
|---------|----------|
| Стадий завершено | Завершённые StageExecutor'ы |
| Проверок проведено | Стадии, где Менеджер — reviewer |
| Проектов одновременно | Макс. одновременных активных |

**ГАП (индивидуальные + шаблонные, но раздельно!):**

| Столбец | Описание |
|---------|----------|
| Проверок проведено | Аналогично СДП |
| Ср. время ревью (дн.) | Аналогично |
| Возвратов на доработку | Аналогично |
| Проектов | CRM-карточки с `gap_id` |

**Чертёжники / Дизайнеры (индивидуальные и шаблонные, раздельно!):**

| Столбец | Описание |
|---------|----------|
| Стадий завершено | Завершённые StageExecutor'ы **данного типа проекта** |
| Площадь (м²) | Суммарная площадь |
| Кол-во правок | Суммарный `revision_count` |
| Проектов одновременно | Макс. одновременных **данного типа** |
| Ср. время выполнения (дн.) | `AVG(completed_date - assigned_date)` |

**Замерщики (только индивидуальные):**

| Столбец | Описание |
|---------|----------|
| Замеров проведено | Завершённые стадии «Замер» |
| Ср. дней на замер | От назначения до завершения |

**ДАН (только авторский надзор):**

| Столбец | Описание |
|---------|----------|
| Надзоров ведёт | `supervision_cards.dan_id` (активные) |
| Визитов на объекты | `SUM(supervision_visits)` |
| Дефектов найдено | `SUM(defects_found)` |
| Дефектов устранено | `SUM(defects_resolved)` |
| % устранения | `resolved / found * 100` |
| Экономия бюджета | `SUM(budget_savings)` |
| Просрочки закупок | Стадии с `actual_date > plan_date` |

### Графики сравнения (под таблицей)

Для каждой роли — 3 графика (данные **только по выбранному типу проекта**):

**Для CRM-проектов (индивидуальные / шаблонные):**

1. **Horizontal Bar — KPI по сотрудникам**
   - Цвет: зелёный > 80%, жёлтый 60-80%, красный < 60%

2. **Grouped Bar — В срок vs Просрочки**
   - 2 серии: в срок (зелёный), с просрочкой (красный)

3. **Line Chart — Динамика KPI по месяцам**
   - Линии: по одной на каждого сотрудника роли

**Для авторского надзора:**

1. **Horizontal Bar — KPI по ДАН/СМ**
2. **Stacked Bar — Дефекты: найдено vs устранено**
3. **Line Chart — Динамика по месяцам**

## 9. Модуль 3: Карточка сотрудника

### Как открывается

- **Руководитель / Ст.менеджер:** Выбирает сотрудника из ComboBox под сравнительной таблицей
- **СДП / ГАП / Менеджер:** Выбирает из списка своих исполнителей, или видит свой профиль
- **Исполнители / Замерщики / ДАН:** Автоматически загружается **только свой** профиль (без выбора)

### Структура карточки (CRM-проекты: индивидуальные / шаблонные)

#### Блок 1: KPI-карточки (6 штук)

| # | Карточка | Описание | Цвет |
|---|----------|----------|------|
| 1 | **KPI** | Интегральная оценка 0-100% (по данному типу проекта) | Зелёный >80, жёлтый 60-80, красный <60 |
| 2 | **Выполнение в срок** | % стадий в срок | Аналогично |
| 3 | **Скорость работы** | Среднее отклонение от нормо-дней | Зелёный <=0, жёлтый 1-3, красный >3 |
| 4 | **Качество** | % стадий без правок (revision_count = 0) | Аналогично |
| 5 | **Нагрузка** | Текущих проектов / рекомендуемый максимум | Зелёный <80%, жёлтый 80-100%, красный >100% |
| 6 | **NPS** | Средняя оценка от клиентов | По шкале 0-10 |

#### Блок 2: Графики (2 ряда по 2)

1. **Динамика KPI по месяцам** (Line Chart) — 12 месяцев, 3 линии: KPI, В срок%, Качество%
2. **Просрочки по стадиям** (Stacked Bar) — по стадиям **данного типа проекта**
3. **Нагрузка по месяцам** (Area Chart) — проекты **данного типа** + пунктир рекомендуемого макс.
4. **Распределение времени** (Donut) — по стадиям **данного типа**

## 10. Модуль 4: Система клиентских опросов через Яндекс Формы

### Почему Яндекс Формы?

1. **Уже есть интеграция с Яндекс** — CRM уже работает с Яндекс.Диском для хранения файлов
2. **Не нужно создавать свою веб-страницу** — Яндекс Формы предоставляют готовый интерфейс
3. **Бесплатно** — для бизнес-аккаунта Яндекс 360
4. **Готовая аналитика** — Яндекс Формы сами показывают сводку ответов
5. **Работает на мобильных** — клиент заполнит с телефона без проблем

### Как это работает (пошагово)

```
ПОДГОТОВКА (один раз):
1. Создаём шаблон анкеты в Яндекс Формах (forms.yandex.ru)
2. Настраиваем webhook: при заполнении формы → POST на наш сервер
3. В шаблоне формы добавляем скрытое поле contract_id

АВТОМАТИЧЕСКИЙ ПРОЦЕСС (при каждом завершении проекта):
1. CRM-карточка переходит в статус «СДАН»
   ↓
2. Сервер создаёт уникальную ссылку на Яндекс Форму
   с предзаполненным contract_id
   ↓
3. Ссылка отправляется клиенту:
   - Через Telegram (если есть telegram у клиента)
   - Через уведомление менеджеру (он отправляет вручную)
   ↓
4. Клиент переходит по ссылке → видит красивую форму Яндекс
   ↓
5. Клиент заполняет и нажимает «Отправить»
   ↓
6. Яндекс Формы отправляет webhook на наш сервер:
   POST /api/surveys/yandex-webhook
   ↓
7. Сервер:
   a. Парсит ответы
   b. Находит contract_id из скрытого поля
   c. Сохраняет в таблицу client_surveys
   d. Привязывает к сотрудникам проекта
   e. Отправляет уведомление: «Клиент оставил отзыв»
   ↓
8. Данные учитываются в KPI сотрудников
```

#### Обработка webhook на сервере

```python
@router.post("/yandex-webhook")
async def handle_yandex_form_webhook(
    request: Request,
    db: Session = Depends(get_db)
):
    """
    Принимает ответы из Яндекс Форм.
    Не требует JWT авторизации (вызывается внешним сервисом).
    Защита: проверка survey_token.
    """
    data = await request.json()
    answers = {a['question_id']: a['value'] for a in data.get('answers', [])}

    # 1. Находим токен и проверяем его валидность
    token = answers.get('hidden_survey_token')
    survey = db.query(ClientSurvey).filter(
        ClientSurvey.access_token == token,
        ClientSurvey.status != 'completed'  # не принимаем повторные
    ).first()

    if not survey:
        raise HTTPException(404, "Опрос не найден или уже заполнен")

    # 2. Сохраняем ответы
    survey.nps_score = int(answers.get('q1_nps', 0))
    survey.csat_score = int(answers.get('q2_csat', 0))
    survey.design_score = int(answers.get('q3_design', 0))
    survey.deadline_score = int(answers.get('q4_deadline', 0))
    survey.communication_score = int(answers.get('q5_communication', 0))
    survey.expectations_score = int(answers.get('q6_expectations', 0))
    survey.supervision_score = int(answers.get('q7_supervision', 0)) if answers.get('q7_supervision') else None
    survey.comment = answers.get('q8_comment', '')
    survey.status = 'completed'
    survey.completed_at = datetime.utcnow()
    survey.yandex_answer_id = data.get('answer_id')

    db.commit()

    # 3. Отправляем уведомление менеджеру
    await notify_survey_completed(db, survey)

    return {"status": "ok"}
```

### Привязка результатов опроса к сотрудникам

Когда опрос заполнен, его результаты привязываются к **всем сотрудникам, работавшим на этом проекте**. При этом учитывается **тип проекта**:

```python
def link_survey_to_employees(db, survey):
    """
    Находит всех сотрудников, работавших над проектом, и
    привязывает к ним результаты опроса.
    """
    contract_id = survey.contract_id

    # Для CRM-проектов (индивидуальных и шаблонных)
    crm_card = db.query(CRMCard).filter(
        CRMCard.contract_id == contract_id
    ).first()

    employees_on_project = set()

    if crm_card:
        # Ответственные по карточке
        if crm_card.senior_manager_id:
            employees_on_project.add(crm_card.senior_manager_id)
        if crm_card.sdp_id:
            employees_on_project.add(crm_card.sdp_id)
        if crm_card.gap_id:
            employees_on_project.add(crm_card.gap_id)
        if crm_card.manager_id:
            employees_on_project.add(crm_card.manager_id)
        if crm_card.surveyor_id:
            employees_on_project.add(crm_card.surveyor_id)

        # Все исполнители стадий
        for executor in crm_card.stage_executors:
            if executor.executor_id:
                employees_on_project.add(executor.executor_id)

    # Для авторского надзора
    supervision = db.query(SupervisionCard).filter(
        SupervisionCard.contract_id == contract_id
    ).first()

    if supervision:
        if supervision.senior_manager_id:
            employees_on_project.add(supervision.senior_manager_id)
        if supervision.dan_id:
            employees_on_project.add(supervision.dan_id)

    return employees_on_project
```

## 11. Источники данных и формулы расчёта KPI

### Общий KPI сотрудника (по типу проекта)

**Важно:** KPI считается **отдельно для каждого типа проекта**. Один сотрудник может иметь разный KPI на индивидуальных и шаблонных проектах.

#### Формула для CRM-проектов (индивидуальные и шаблонные)

```
KPI = (W1 × Kсрок) + (W2 × Kкачество) + (W3 × Kскорость) + (W4 × Knps)
```

Где:
- `Kсрок` — коэффициент выполнения в срок (0-100%)
- `Kкачество` — коэффициент качества (0-100%)
- `Kскорость` — коэффициент скорости работы (0-100%)
- `Knps` — коэффициент клиентской оценки (0-100%)

#### Формула для авторского надзора

```
KPI_надзор = (W1 × Kзакупки) + (W2 × Kдефекты) + (W3 × Kвизиты) + (W4 × Knps)
```

Где:
- `Kзакупки` — % стадий закупок, завершённых до plan_date
- `Kдефекты` — % устранённых дефектов
- `Kвизиты` — соответствие плану визитов (если есть норматив)
- `Knps` — клиентская оценка надзора

#### Веса компонентов по ролям

**Для CRM-проектов (индивидуальные и шаблонные):**

| Роль | W1 (Срок) | W2 (Качество) | W3 (Скорость) | W4 (NPS) |
|------|-----------|---------------|----------------|----------|
| Старший менеджер | 0.30 | 0.20 | 0.15 | 0.35 |
| Менеджер | 0.30 | 0.20 | 0.20 | 0.30 |
| СДП | 0.25 | 0.35 | 0.25 | 0.15 |
| ГАП | 0.25 | 0.35 | 0.25 | 0.15 |
| Чертёжник | 0.30 | 0.35 | 0.25 | 0.10 |
| Дизайнер | 0.25 | 0.35 | 0.25 | 0.15 |
| Замерщик | 0.35 | 0.15 | 0.35 | 0.15 |

**Для авторского надзора:**

| Роль | W1 (Закупки в срок) | W2 (Дефекты) | W3 (Визиты) | W4 (NPS) |
|------|---------------------|--------------|-------------|----------|
| Старший менеджер | 0.30 | 0.20 | 0.15 | 0.35 |
| ДАН | 0.30 | 0.30 | 0.20 | 0.20 |

**Логика весов:**
- Для **исполнителей** (чертёжники, дизайнеры) важнее всего качество и сроки
- Для **менеджеров** и **старших менеджеров** важнее обратная связь от клиентов
- Для **замерщиков** критична скорость (клиент ждёт)
- Для **проверяющих** (СДП, ГАП) — качество проверки
- Для **ДАН** — контроль дефектов и сроки закупок

### Kсрок — Коэффициент выполнения в срок

```python
def calc_k_deadline(employee_id, project_type, period_start, period_end):
    """
    Считает процент стадий, завершённых до дедлайна.

    Данные фильтруются по типу проекта через JOIN contracts.
    """
    stages = db.query(StageExecutor).join(
        CRMCard, StageExecutor.crm_card_id == CRMCard.id
    ).join(
        Contract, CRMCard.contract_id == Contract.id
    ).filter(
        Contract.project_type == project_type,  # <-- ФИЛЬТР ПО ТИПУ!
        StageExecutor.executor_id == employee_id,
        StageExecutor.completed == True,
        StageExecutor.completed_date.between(period_start, period_end),
        StageExecutor.deadline.isnot(None)
    ).all()

    if not stages:
        return 100  # нет данных = нет проблем

    on_time = sum(1 for s in stages
                  if s.completed_date.date() <= parse_date(s.deadline))

    return (on_time / len(stages)) * 100
```

### Kкачество — Коэффициент качества

```python
def calc_k_quality(employee_id, project_type, period_start, period_end):
    """
    Процент стадий, принятых без правок (revision_count = 0).
    """
    stages = get_completed_stages(employee_id, project_type, period_start, period_end)

    if not stages:
        return 100

    no_revisions = 0
    for stage in stages:
        workflow = db.query(StageWorkflowState).filter(
            StageWorkflowState.crm_card_id == stage.crm_card_id,
            StageWorkflowState.stage_name == stage.stage_name
        ).first()

        if not workflow or workflow.revision_count == 0:
            no_revisions += 1

    return (no_revisions / len(stages)) * 100
```

### Kскорость — Коэффициент скорости

```python
def calc_k_speed(employee_id, project_type, period_start, period_end):
    """
    Сравнивает фактическое время выполнения с нормо-днями.
    """
    stages = get_completed_stages(employee_id, project_type, period_start, period_end)

    if not stages:
        return 100

    speed_scores = []
    for stage in stages:
        actual_days = count_business_days(
            stage.assigned_date.date(),
            stage.completed_date.date()
        )

        timeline = db.query(ProjectTimelineEntry).filter(
            ProjectTimelineEntry.contract_id == stage.crm_card.contract_id,
            ProjectTimelineEntry.executor_role == get_executor_role(employee_id)
        ).first()

        norm_days = timeline.norm_days if timeline else actual_days

        if norm_days > 0 and actual_days > 0:
            ratio = min(norm_days / actual_days, 1.2)
            speed_scores.append(ratio * 100)

    return sum(speed_scores) / len(speed_scores) if speed_scores else 100
```

### Kзакупки — Коэффициент закупок в срок (только авторский надзор)

```python
def calc_k_procurement(employee_id, period_start, period_end):
    """
    Процент стадий закупок, завершённых до plan_date.
    Только для ДАН и старших менеджеров на надзоре.
    """
    # Находим все карточки надзора, где сотрудник — ДАН или СМ
    supervision_cards = db.query(SupervisionCard).filter(
        or_(
            SupervisionCard.dan_id == employee_id,
            SupervisionCard.senior_manager_id == employee_id
        )
    ).all()

    card_ids = [c.id for c in supervision_cards]

    entries = db.query(SupervisionTimelineEntry).filter(
        SupervisionTimelineEntry.supervision_card_id.in_(card_ids),
        SupervisionTimelineEntry.actual_date.isnot(None),
        SupervisionTimelineEntry.plan_date.isnot(None)
    ).all()

    if not entries:
        return 100

    on_time = sum(1 for e in entries
                  if parse_date(e.actual_date) <= parse_date(e.plan_date))

    return (on_time / len(entries)) * 100
```

### Knps — Коэффициент клиентской оценки

```python
def calc_k_nps(employee_id, project_type, period_start, period_end):
    """
    Средний NPS по проектам сотрудника ДАННОГО ТИПА.
    """
    if project_type in ('Индивидуальный', 'Шаблонный'):
        surveys = db.query(ClientSurvey).join(
            Contract, ClientSurvey.contract_id == Contract.id
        ).join(
            CRMCard, CRMCard.contract_id == Contract.id
        ).filter(
            Contract.project_type == project_type,
            ClientSurvey.status == 'completed',
            ClientSurvey.completed_at.between(period_start, period_end),
            # Сотрудник участвовал в проекте
            or_(
                CRMCard.senior_manager_id == employee_id,
                CRMCard.sdp_id == employee_id,
                CRMCard.gap_id == employee_id,
                CRMCard.manager_id == employee_id,
                CRMCard.surveyor_id == employee_id,
            )
        ).all()

        # Также через stage_executors
        surveys_via_executor = db.query(ClientSurvey).join(
            Contract, ClientSurvey.contract_id == Contract.id
        ).join(
            CRMCard, CRMCard.contract_id == Contract.id
        ).join(
            StageExecutor, StageExecutor.crm_card_id == CRMCard.id
        ).filter(
            Contract.project_type == project_type,
            StageExecutor.executor_id == employee_id,
            ClientSurvey.status == 'completed',
        ).all()

        all_surveys = list(set(surveys + surveys_via_executor))

    else:  # Авторский надзор
        all_surveys = db.query(ClientSurvey).join(
            Contract, ClientSurvey.contract_id == Contract.id
        ).join(
            SupervisionCard, SupervisionCard.contract_id == Contract.id
        ).filter(
            ClientSurvey.project_type == 'supervision',
            ClientSurvey.status == 'completed',
            or_(
                SupervisionCard.dan_id == employee_id,
                SupervisionCard.senior_manager_id == employee_id,
            )
        ).all()

    if not all_surveys:
        return None  # Нет данных — не учитываем в KPI

    nps_scores = [s.nps_score * 10 for s in all_surveys if s.nps_score is not None]

    return sum(nps_scores) / len(nps_scores) if nps_scores else None
```

**Если `Knps = None` (нет опросов):** вес NPS перераспределяется пропорционально:

```python
def calc_total_kpi(employee_id, project_type, period_start, period_end, position):
    weights = get_weights(position, project_type)

    k_deadline = calc_k_deadline(employee_id, project_type, period_start, period_end)
    k_quality = calc_k_quality(employee_id, project_type, period_start, period_end)
    k_speed = calc_k_speed(employee_id, project_type, period_start, period_end)
    k_nps = calc_k_nps(employee_id, project_type, period_start, period_end)

    if k_nps is None:
        # Перераспределяем вес NPS
        remaining = 1.0 - weights['nps']
        return (
            (weights['deadline'] / remaining) * k_deadline +
            (weights['quality'] / remaining) * k_quality +
            (weights['speed'] / remaining) * k_speed
        )
    else:
        return (
            weights['deadline'] * k_deadline +
            weights['quality'] * k_quality +
            weights['speed'] * k_speed +
            weights['nps'] * k_nps
        )
```

## 16. Риски и ограничения

### Технические риски

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Нехватка данных для KPI (мало завершённых стадий) | Высокая для новых | Показывать «Недостаточно данных» если < 3 стадий |
| Медленный расчёт KPI (много JOIN) | Средняя | Кеш + ежедневные снимки в `employee_kpi_snapshots` |
| Клиенты не заполняют опросы | Высокая | Напоминания, простой Яндекс интерфейс, вес NPS = 0 если нет данных |
| Webhook Яндекс Форм ненадёжен | Низкая | Ручной импорт через CSV (Яндекс позволяет скачать ответы) |
| Права доступа обходятся через API | Средняя | Серверная фильтрация (не клиентская) + тесты |
| deadline — String, не Date | Известный баг | `CAST(deadline AS DATE)` (уже решено) |

### Бизнес-ограничения

1. **Конфиденциальность зарплат:** Видны только Руководителю и Старшему менеджеру (своей команды)
2. **Мотивация:** Исполнители видят только свой KPI, не чужой — это мотивирует, а не демотивирует
3. **NPS защита от накрутки:** Один опрос на один договор, уникальный токен
4. **Яндекс Формы:** Нужен бизнес-аккаунт Яндекс 360 для webhook

### Зависимости

- **Яндекс Формы:** Нужен доступ к Яндекс 360, настройка webhook, публичный URL сервера для приёма webhook
- **Тренды KPI:** Первые тренды появятся через 2-3 месяца после запуска снимков
- **Нормо-дни:** Точность `Kскорость` зависит от заполненности `project_timeline_entries.norm_days`

## Приложение В: Примеры SQL-запросов

### KPI по типу проекта — стадии в срок

```sql
SELECT
    e.full_name,
    e.position,
    c.project_type,
    COUNT(*) AS total_completed,
    COUNT(*) FILTER (
        WHERE se.completed_date <= CAST(se.deadline AS TIMESTAMP)
    ) AS on_time,
    ROUND(
        COUNT(*) FILTER (
            WHERE se.completed_date <= CAST(se.deadline AS TIMESTAMP)
        )::DECIMAL / COUNT(*) * 100, 1
    ) AS on_time_pct
FROM stage_executors se
JOIN employees e ON e.id = se.executor_id
JOIN crm_cards cc ON cc.id = se.crm_card_id
JOIN contracts c ON c.id = cc.contract_id
WHERE se.completed = TRUE
  AND se.deadline IS NOT NULL
  AND c.project_type = 'Индивидуальный'  -- <-- ФИЛЬТР!
  AND EXTRACT(YEAR FROM se.completed_date) = 2026
GROUP BY e.id, e.full_name, e.position, c.project_type
ORDER BY on_time_pct DESC;
```

```sql
    UNION SELECT gap_id FROM crm_cards WHERE senior_manager_id = :my_id AND gap_id IS NOT NULL
    UNION SELECT manager_id FROM crm_cards WHERE senior_manager_id = :my_id AND manager_id IS NOT NULL
    UNION SELECT surveyor_id FROM crm_cards WHERE senior_manager_id = :my_id AND surveyor_id IS NOT NULL
    -- Исполнители стадий
    UNION SELECT se.executor_id FROM stage_executors se
        JOIN crm_cards cc ON cc.id = se.crm_card_id
        WHERE cc.senior_manager_id = :my_id AND se.executor_id IS NOT NULL
    -- ДАН по надзору
    UNION SELECT sv.dan_id FROM supervision_cards sv
        WHERE sv.senior_manager_id = :my_id AND sv.dan_id IS NOT NULL
) AS visible;
```

### Метрики авторского надзора по ДАН

```sql
SELECT
    e.full_name,
    COUNT(DISTINCT sv.id) AS total_supervisions,
    SUM(ste.defects_found) AS total_defects_found,
    SUM(ste.defects_resolved) AS total_defects_resolved,
    ROUND(
        SUM(ste.defects_resolved)::DECIMAL /
        NULLIF(SUM(ste.defects_found), 0) * 100, 1
    ) AS defect_resolve_rate,
    SUM(ste.budget_savings) AS total_savings,
    (SELECT COUNT(*) FROM supervision_visits svv
     JOIN supervision_cards svc ON svc.id = svv.supervision_card_id
     WHERE svc.dan_id = e.id) AS total_visits
FROM employees e
JOIN supervision_cards sv ON sv.dan_id = e.id
JOIN supervision_timeline_entries ste ON ste.supervision_card_id = sv.id
WHERE e.position IN ('ДАН', 'Дизайнер авторского надзора')
GROUP BY e.id, e.full_name
ORDER BY total_supervisions DESC;
```
