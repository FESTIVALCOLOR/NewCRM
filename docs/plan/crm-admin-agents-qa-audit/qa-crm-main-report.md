## 2. Таблица сроков (norm_days_templates)

### Механизм загрузки

**Файл:** `server/services/timeline_service.py`

Приоритет нормо-дней:
1. Кастомный шаблон для конкретного агента (`agent_type = 'ФЕСТИВАЛЬ'` или `'ПЕТРОВИЧ'`)
2. Кастомный шаблон для `'Все агенты'`
3. Формулы (хардкод) — fallback

### Анализ по вариациям

#### Индивидуальные (вариации 1-6): `build_project_timeline_template()`

| Подтип | Стадии в таймлайне | Фильтрация |
|--------|-------------------|------------|
| Полный (с 3д визуализацией) | START + STAGE1 + STAGE2 + STAGE3 + STAGE4 | Все этапы |
| Эскизный (с коллажами) | START + STAGE1 + STAGE2(Подэтап 2.1) + STAGE3 | `'Эскизный' in project_subtype` |
| Планировочный | START + STAGE1 | `'Планировочный' in project_subtype` |

**Проблема Б2 (средняя):** При подтипе "Планировочный" таймлайн содержит только STAGE1. Однако CRM-карточка может перейти в Стадия 2, 3, 4 — нормо-дней для этих стадий не будет, поскольку taймлайн был сгенерирован только для STAGE1. Если пользователь откроет таблицу сроков после перехода на Стадию 2 — она будет пустой для стадий 2-4.

#### Коэффициент площади (К)

```python
K = calc_area_coefficient(area)  # = max(0, int((area - 1) // 100))
# area=100 → K=0, area=200 → K=1, area=300 → K=2
```

Нормо-дни рассчитываются по формуле: `raw_norm_days = base + K * multiplier`. Далее пропорциональное распределение в пределах `contract_term`.

#### Шаблонные (вариации 7-14): `build_template_project_timeline()`

| Подтип | Стадии | Срок (area<=90) |
|--------|--------|----------------|
| Стандарт | STAGE1 + STAGE2 | 20 рабочих дней |
| Стандарт с визуализацией | STAGE1 + STAGE2 + STAGE3 | 20+25=45 рабочих дней |
| Проект ванной комнаты | STAGE1 + STAGE2 | 10 рабочих дней |
| Проект ванной комнаты с визуализацией | STAGE1 + STAGE2 + STAGE3 | 10+20=30 рабочих дней |

**Проблема Б3 (средняя):** Для шаблонных проектов в `build_template_project_timeline()` коэффициент K жёстко равен 0 (строка 445: `return entries, contract_term, 0`), при этом кастомный шаблон применяется через: `e['raw_norm_days'] = c.base_norm_days + 0 * c.k_multiplier` (строка 409). Поле `k_multiplier` в NormDaysTemplate хранится, но для шаблонных проектов никогда не применяется — это потенциальная путаница для администраторов при настройке.

### NormDaysTemplate — хранение в БД

**Файл:** `server/routers/norm_days_router.py`

Кастомные шаблоны хранятся в таблице `NormDaysTemplate` с ключами:
- `project_type` (Индивидуальный / Шаблонный)
- `project_subtype` (точная строка подтипа)
- `agent_type` (ФЕСТИВАЛЬ / ПЕТРОВИЧ / Все агенты)

**Важно:** Если кастомного шаблона нет, система автоматически использует формулы. Это значит, что для всех 14 вариаций нормо-дни будут работать из коробки без заполнения БД.

## 6. Дашборды

**Endpoint:** `GET /api/dashboard/crm?project_type=...`
**Файл:** `server/routers/dashboard_router.py`, строки 162–223

### Что считается

```python
# total_orders = все договоры с нужным project_type/status
# active_orders = db.query(CRMCard).count()   ← ВСЕ карточки, без фильтра!
# archive_orders = total_orders - active_orders (если разница > 0, иначе 0)
```

**Проблема К1 (критическая):** `active_orders = db.query(crm_table).count()` считает ВСЕ CRM-карточки без фильтрации по `project_type`. Если в базе есть и Индивидуальные, и Шаблонные карточки, то при запросе дашборда для `project_type='Индивидуальный'` в `active_orders` попадут также шаблонные карточки. Это приводит к неверным числам в виджетах дашборда.

**Корректный запрос должен быть:**
```python
active_orders = db.query(CRMCard).join(Contract).filter(contract_condition).count()
```

**Проблема К2 (критическая):** `archive_orders` рассчитывается как разность `total_orders - active_orders`. Если `active_orders` завышен (из-за К1), то `archive_orders = max(0, ...)` может быть равен 0 даже когда архивные договоры есть. Кроме того, логика "архив = все договоры минус активные CRM-карточки" семантически некорректна: договор переходит в архив через изменение `Contract.status`, а не через удаление CRM карточки.

## 11. Автозаполнение дат

**Клиентский файл:** `utils/timeline_calc.py`
**Серверный файл:** `server/services/timeline_service.py`

### Алгоритм `calc_planned_dates(entries)`

```
1. Найти START entry → prev_date = START.actual_date
2. Для каждого entry (кроме header):
   - norm = entry.norm_days (или custom_norm_days если задан)
   - Если prev_date и norm > 0: planned_date = add_working_days(prev_date, norm)
   - Если prev_date и norm == 0: planned_date = prev_date (наследует)
   - Иначе: planned_date = ''
   - Обновить prev_date: actual_date если есть, иначе planned_date
```

### Алгоритм расчёта нормо-дней (Timeline Service)

1. `K = max(0, int((area - 1) // 100))` — коэффициент площади
2. `contract_term` — из формул по подтипу и площади
3. `raw_norm_days = base + K * multiplier` для каждого подэтапа
4. Пропорциональный расчёт: `norm_days[i] = round(cumulative_i / total_raw * contract_term) - round(cumulative_{i-1} / ...)`
5. Корректировка последнего элемента, чтобы сумма = contract_term

**Работает для всех 14 вариаций.** Различие только в наборе подэтапов (фильтрация) и значениях K/contract_term.

**Проблема Н3 (низкая):** В crm_router.py строки 1234-1247 используется отдельный механизм расчёта дедлайна для автозаполнения при перемещении карточки: берётся `norm_days` последнего подэтапа текущей стадии и прибавляется через `_add_business_days`. Это независимый расчёт от `calc_planned_dates`. При несинхронизированных данных (разные norm_days в timeline и в карточке) дедлайны могут расходиться.

При наличии кастомного шаблона: применяется через `base_norm_days + K * k_multiplier` (для шаблонных K=0).
