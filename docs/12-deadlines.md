# Система дедлайнов

> Дедлайны проектов, таймлайны, рабочие дни, workflow исполнителей.

## Полная спецификация (бизнес-требования)

### 1. Дедлайн проекта (общий)

Указывается в карточке CRM «Исполнители и дедлайн».
Определяется исходя из:
- **Срок договора** (в карточке договора, пр. 60 раб. дней)
- **Дата начала разработки** (из таблицы сроков CRM)

```
Дедлайн проекта = Дата начала разработки + Срок договора (рабочие дни)
```

### 2. Дата начала разработки (START)

```python
START = max(contract_date, survey_date, tech_task_date, advance_payment_paid_date)
# contract_date — дата подписания договора
# survey_date — дата замера
# tech_task_date — дата технического задания
# advance_payment_paid_date — дата 1-го платежа (Аванс)
```

START определяется **автоматически**, в настройках norm_days **не выставляется**, исполнителя у неё нет.

### 3. Нормо-дни (norm_days)

Определяются по формуле в администрировании для каждого:
- **Типа проекта** (Индивидуальный / Шаблонный)
- **Подтипа проекта** (Стандарт, Полный, Эскизный, Планировочный и т.д.)
- **Площади**
- **Типа агента** (Все агенты / Фестиваль / Петрович и т.д.)

**Приоритет:** При наличии индивидуального шаблона для агента — используется он, иначе fallback на «Все агенты».

### 4. Автоматизация дат в таблице сроков

**Принцип:** Даты заполняются автоматически при workflow-действиях (кнопки на карточке CRM).

#### Пример: Этап 1 → Подэтап 1.1

1. СДП перемещает карточку в Kanban → «Стадия 1: Планировочные решения»
2. Открывается окно выбора исполнителя + дедлайн:
   - Дата предлагается: `START.actual_date + norm_days[подэтап]`
   - СДП может согласиться или **выставить свою** (бОльшую)
3. **Если СДП выставил свою дату:**
   - В таблице сроков: `~~4~~` (зачёркнуто) и рядом **красным** новая норма дней
   - Tooltip: «Превышение стандартного значения нормо-дней. Учесть в дальнейших стадиях»
4. Исполнитель работает → нажимает «Сдать работу»:
   - В таблице сроков заполняется actual_date для «Разработка 3 вар. планировок»
5. СДП проверяет — 2 варианта:
   - **Вар.1 (На исправления):** дата «Проверка СДП» → карточка к исполнителю → «Сдать работу» → дата «Правка чертежником» → повторная проверка
   - **Вар.2 (Согласовать с клиентом):** промежуточные даты пропускаются → сразу «Отправка клиенту»
6. Клиент согласовал → СДП **вручную** вставляет дату «Отправка клиенту»
7. Следующая дата считается от даты «Отправка клиенту»

#### Сбор правок от клиента

Автоматический расчёт: от даты «Отправка клиенту» до назначения следующего подэтапа.

### 5. Ручной ввод дат

**Вручную вводятся только:**
1. Даты согласования с клиентом
2. Даты при назначении исполнителей (из диалога назначения, с автопредложением)

**Все остальные даты** заполняются автоматически при workflow-действиях.

### 6. Кнопка-карандаш (ручное редактирование)

- Рядом с каждой ячейкой даты — кнопка с иконкой карандаша (edit.svg)
- По клику — ячейка переключается в режим QDateEdit
- При изменении даты — ВСЕ последующие пересчитываются каскадно

### 7. Отображение дат в таблице

- **Пустая ячейка** — стадия не завершена (planned_date только в tooltip при наведении)
- **Зелёный фон (#E8F5E9)** — actual_date заполнена (факт)
- **Tooltip** — показывает планируемую дату и/или сравнение «план vs факт»

### 8. Синхронизация norm_days

- **Односторонняя:** из администрирования → в карточку (не наоборот)
- **Исключение:** Если при назначении исполнителя СДП/ГАП/менеджер ставит бОльшую норму дней → она сохраняется (не перезаписывается шаблоном)
- **Отображение превышения:** зачёркнутая стандартная + красная кастомная

## Типы дедлайнов (технический)

### 1. Дедлайн карточки CRM

```sql
crm_cards.deadline — общий дедлайн проекта
```

Устанавливается менеджером, отображается на Kanban карточке.

### 2. Дедлайн согласования

```sql
crm_cards.approval_deadline — дедлайн согласования с клиентом
```

### 3. Дедлайн исполнителя

```sql
stage_executors.deadline — персональный дедлайн исполнителя на стадию
```

Рассчитывается автоматически:
```python
# База: actual_date предыдущего подэтапа (если есть), иначе today
base_date = prev_actual_date or today
deadline = base_date + norm_days (рабочие дни, Пн-Пт)
```

### 4. Даты в таймлайне

```sql
project_timeline_entries.actual_date — фактическая дата завершения этапа
```

### 5. Планируемые даты (planned dates)

```python
# Рассчитываются цепочкой в utils/timeline_calc.py → calc_planned_dates()
planned[START] = START.actual_date
planned[N] = prev_date + norm_days[N]
# prev_date = actual_date (если заполнена) или planned_date предыдущего
```

Отображаются **ТОЛЬКО в tooltip** при наведении мышки (не в ячейке).
Фактические даты (actual_date) — зелёный фон (`#E8F5E9`).
Ячейки заблокированы (read-only QLabel), редактирование через кнопку-карандаш (edit.svg).

### 6. Каскадный пересчёт

При ручном изменении любой даты в таймлайне:
```
_on_date_changed() → _recalculate_days() → _populate_table() → _calc_planned_dates()
```
Все planned_date пересчитываются автоматически, tooltip обновляются.

**Тесты:** `tests/client/test_planned_dates.py` — 13 тестов (цепочка, каскад, кросс-стадия, edge cases)

### Формулы расчёта

#### Коэффициент площади
```python
K = max(0, int((area - 1) // 100))
# area=50  → K=0
# area=120 → K=1
# area=250 → K=2
```

#### Базовые дни (raw_norm_days)
Каждый этап имеет базовое количество дней, зависящее от K.

#### Нормативные дни (norm_days)
```python
# Пропорциональное распределение по сроку договора
norm_days = contract_term * (raw_norm_days / sum(in_scope_raw_norm_days))

# contract_term — срок договора (все рабочие дни)
# in_scope — только этапы, входящие в scope подтипа
```

#### Фактические дни (actual_days)
```python
actual_days = networkdays(prev_date, current_date)
# Рабочие дни между последовательными actual_dates
```

### Этапы индивидуальных проектов

```
START           → ДАТА НАЧАЛА РАЗРАБОТКИ

STAGE1          → ЭТАП 1: ПЛАНИРОВОЧНОЕ РЕШЕНИЕ
  Подэтап 1.1  → Разработка 3 вар. планировок → Проверка СДП → Правка → Клиент
  Подэтап 1.2  → Финальное план. решение (1 круг правок)
  Подэтап 1.3  → Финальное план. решение (2 круг правок)

STAGE2          → ЭТАП 2: КОНЦЕПЦИЯ ДИЗАЙНА (Полный + Эскизный)
  Подэтап 2.1  → Мудборды
  Подэтап 2.2  → Визуализация 1 помещения (только Полный)
  Подэтап 2.3-2.4 → Правки визуализации (1, 2 круг)
  Подэтап 2.5  → Визуализация остальных помещений
  Подэтап 2.6-2.7 → Правки всех визуализаций (1, 2 круг)

STAGE3          → ЭТАП 3: РАБОЧАЯ ДОКУМЕНТАЦИЯ (Полный + Эскизный)
  12 шагов     → Подготовка → Разработка РД → Проверки ГАП → Клиент → Закрытие
```

### Этапы шаблонных проектов

```
START   → ДАТА НАЧАЛА РАЗРАБОТКИ

T1      → СТАДИЯ 1: ПЛАНИРОВОЧНЫЕ РЕШЕНИЯ
  T1_1  → Разработка 3 вар. → проверка Менеджером → клиент
  T1_2  → Финальное план. решение

T2      → СТАДИЯ 2: РАБОЧИЕ ЧЕРТЕЖИ
  12 шагов → Подготовка → Разработка → ГАП → Клиент → Закрытие

T3      → СТАДИЯ 3: 3Д ВИЗУАЛИЗАЦИЯ (только если подтип с визуализацией)
  6 шагов → Разработка виз → проверка Менеджером → клиент → закрытие
```
