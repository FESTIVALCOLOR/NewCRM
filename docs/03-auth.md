# Авторизация и права доступа

> Система авторизации, JWT токены, роли и разграничение прав.

## Архитектура авторизации

```
┌─────────────┐     POST /api/auth/login      ┌──────────────┐
│  LoginWindow │ ──────────────────────────── → │  server/auth  │
│  (PyQt5)    │ ← ──────────────────────────── │  (FastAPI)    │
│             │     JWT token + employee_data   │              │
└─────────────┘                                 └──────────────┘
       │                                               │
       ▼                                               ▼
  MainWindow                                     PostgreSQL
  + SyncManager                                  employees table
  + DataAccess                                   bcrypt hashes
```

## Потоки авторизации

### Сетевой режим (MULTI_USER_MODE=True)

```
1. Пользователь вводит login + password
2. LoginWindow → APIClient.authenticate(login, password)
3. APIClient → POST /api/auth/login {login, password}
4. Сервер: bcrypt.checkpw(password, hash) → JWT token
5. Ответ: {token, employee: {id, full_name, role, position, ...}}
6. APIClient сохраняет token в headers: Authorization: Bearer <token>
7. LoginWindow → MainWindow(employee_data, api_client)
```

### Автономный режим (MULTI_USER_MODE=False)

```
1. Пользователь вводит login + password
2. LoginWindow → DatabaseManager.verify_credentials(login, password)
3. SQLite: SELECT * FROM employees WHERE login = ?
4. bcrypt.checkpw(password, stored_hash)
5. LoginWindow → MainWindow(employee_data, api_client=None)
6. MainWindow работает без SyncManager
```

## JWT токены

### Генерация (server/auth.py)

```python
# Библиотека: PyJWT
# Алгоритм: HS256
# Время жизни: настраиваемое (по умолчанию — сессия)

payload = {
    "sub": employee_id,
    "login": login,
    "role": role,
    "exp": datetime.utcnow() + timedelta(hours=24)
}
token = jwt.encode(payload, SECRET_KEY, algorithm="HS256")
```

### Валидация (server/auth.py)

```python
# Middleware проверяет Authorization header
# При невалидном токене → 401 Unauthorized
# При истекшем токене → 401 с сообщением "Token expired"
```

### Хранение пароля

```python
# Библиотека: bcrypt==4.2.1
# Хэширование при создании сотрудника
import bcrypt
hashed = bcrypt.hashpw(password.encode('utf-8'), bcrypt.gensalt())

# Проверка при входе
bcrypt.checkpw(password.encode('utf-8'), stored_hash)
```

## Роли и права доступа

### Уровни доступа

| Уровень | Описание |
|---------|----------|
| **Admin** | Полный доступ ко всем функциям системы |
| **Manager** | Управление проектами, CRM, статистика, сотрудники |
| **Worker** | Ограниченный доступ — только свои задачи |

### Матрица ролей

| Роль | Табы | Может редактировать | Уровень |
|------|------|---------------------|---------|
| Руководитель студии | Все | Да | Admin |
| Старший менеджер проектов | Все | Да | Admin |
| СДП | CRM, Статистика, Сотрудники | Да | Manager |
| ГАП | CRM, Статистика, Сотрудники | Да | Manager |
| **Менеджер** | CRM, CRM надзора, Статистика, Сотрудники | Да | Manager |
| Дизайнер | CRM (просмотр) | Нет | Worker |
| Чертёжник | CRM (просмотр) | Нет | Worker |
| Замерщик | CRM (просмотр) | Нет | Worker |
| ДАН | CRM надзора | Нет | Worker |

> **Менеджер** — роль аналогичная СДП и ГАП, но с дополнительным доступом к CRM надзора.

### Двойные роли (secondary_position)

Сотрудник может иметь **основную** (`position`) и **дополнительную** (`secondary_position`) должность. Уровень доступа определяется по **старшей** из двух ролей.

**Правила определения уровня:**

| Основная | Дополнительная | Итоговый уровень | Пояснение |
|----------|---------------|------------------|-----------|
| СДП | Дизайнер | Manager | СДП старше → Manager |
| ГАП | Чертёжник | Manager | ГАП старше → Manager |
| Менеджер | Дизайнер | Manager | Менеджер старше → Manager |
| Дизайнер | Чертёжник | Worker | Обе роли одного уровня → Worker |
| Чертёжник | Дизайнер | Worker | Обе роли одного уровня → Worker |
| Дизайнер | — | Worker | Одна роль → по умолчанию |

**Реализация в коде:**
- `_emp_has_pos(employee, *positions)` — **ГРАНТ** доступа: True если хотя бы одна из должностей (основная или дополнительная) совпадает. Используется для предоставления прав.
- `_emp_only_pos(employee, *positions)` — **ОГРАНИЧЕНИЕ**: True только если ВСЕ должности сотрудника входят в указанный набор. Используется для блокировки — если есть вторая, более старшая роль, ограничение снимается.

```python
# Пример: скрыть кнопку только для "чистых" исполнителей
if not _emp_only_pos(employee, 'Дизайнер', 'Чертёжник', 'Замерщик'):
    show_button()  # СДП+Дизайнер увидит, потому что СДП не в списке ограничений
```

## Схема: employees (таблица сотрудников)

```sql
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    full_name VARCHAR(255) NOT NULL,
    phone VARCHAR(50),
    email VARCHAR(255),
    address TEXT,
    birth_date DATE,
    status VARCHAR(20) DEFAULT 'active',
    position VARCHAR(100),        -- основная должность
    secondary_position VARCHAR(100), -- вторичная должность
    department VARCHAR(100),
    login VARCHAR(100) UNIQUE,
    password VARCHAR(255),        -- bcrypt hash
    role VARCHAR(100),            -- определяет права доступа
    created_at TIMESTAMP DEFAULT NOW()
);
```
